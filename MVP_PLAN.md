# План реализации MVP Exocortex

## Обзор
План разбит на 5 этапов, каждый из которых создаёт законченный функциональный блок. После каждого этапа система должна быть работоспособна и готова к тестированию.

---

## Этап 1: Базовая инфраструктура и хранение знаний
**Цель**: Создать фундамент системы — хранилище графа знаний и базовую структуру проекта.

### Задачи:
1. [ ] Настроить структуру проекта (модульная архитектура)
2. [ ] Выбрать и интегрировать графовую БД (Neo4j / NetworkX для начала)
3. [ ] Реализовать модели данных:
   - Узел (Node): id, тип, содержание, метаданные, параметры памяти (strength, decay_rate, last_interacted)
   - Связь (Edge): id, тип, источник, цель, вес, метаданные
4. [ ] Реализовать базовый репозиторий для операций с графом (CRUD)
5. [ ] Добавить миграции/инициализацию БД
6. [ ] Написать тесты на базовые операции с графом

**Критерий готовности**: Можно программно создавать узлы, связи, выполнять простые запросы к графу. Данные сохраняются между запусками.

**Ожидаемое время**: 3-5 дней

---

## Этап 2: Захват знаний и LLM-пайплайн
**Цель**: Реализовать механизм добавления знаний в систему с автоматическим извлечением структуры.

### Задачи:
1. [ ] Интегрировать LLM API (OpenAI / альтернатива)
2. [ ] Реализовать пайплайн извлечения сущностей из текста:
   - Промпт для entity extraction
   - Промпт для relation extraction
   - Парсинг ответов LLM
3. [ ] Создать простой интерфейс для ручного добавления контента:
   - CLI или минимальный веб-интерфейс
   - Возможность сохранить фрагмент текста
4. [ ] Автоматическое создание узлов и связей при добавлении контента
5. [ ] Сохранение исходного контекста (источник, дата, полный текст)
6. [ ] Базовая визуализация: список узлов, простые связи
7. [ ] Тесты на пайплайн извлечения

**Критерий готовности**: Пользователь может добавить текст → система автоматически создаёт узлы и связи в графе → можно увидеть результат.

**Ожидаемое время**: 5-7 дней

---

## Этап 3: Проактивный агент
**Цель**: Реализовать фоновый анализ графа и генерацию инсайтов.

### Задачи:
1. [ ] Реализовать планировщик задач (APScheduler / Celery)
2. [ ] Алгоритм поиска противоречий:
   - Промпт для contradiction detection
   - Выбор кандидатов (узлы с похожими темами но разным содержанием)
3. [ ] Алгоритм поиска неочевидных связей:
   - Векторные эмбеддинги для семантического поиска
   - Поиск кластеров без явных связей
4. [ ] Алгоритм определения «забываемого» контента:
   - Расчёт актуальности на основе decay_rate и last_interacted
   - Выборка кандидатов на напоминание
5. [ ] Генерация дайджеста:
   - Форматирование 1-3 инсайтов
   - Приоритизация по релевантности
6. [ ] Логирование и отладка работы агента
7. [ ] Тесты на алгоритмы анализа

**Критерий готовности**: Агент автоматически запускается, анализирует граф, генерирует дайджест с инсайтами трёх типов.

**Ожидаемое время**: 7-10 дней

---

## Этап 4: Контур взаимодействия и персонализация
**Цель**: Дать пользователю возможность реагировать на инсайты и настроить систему под себя.

### Задачи:
1. [ ] Реализовать inbox для инсайтов (веб-интерфейс или email)
2. [ ] Интерфейс реакции на инсайты:
   - Для противоречий: выбрать сторону / пометить разрешённым / сохранить оба
   - Для связей: подтвердить / отклонить / уточнить
   - Для напоминаний: отметить как полезное / игнорировать
3. [ ] Обновление графа на основе реакций пользователя
4. [ ] Сбор статистики взаимодействий для персонализации
5. [ ] Базовая модель интересов:
   - Подсчёт частоты обращений к темам
   - Корректировка приоритетов тем
6. [ ] Адаптация стиля сообщений агента
7. [ ] Тесты на обновление графа и персонализацию

**Критерий готовности**: Пользователь получает инсайты → реагирует на них → система обновляет граф и подстраивается под паттерны поведения.

**Ожидаемое время**: 5-7 дней

---

## Этап 5: Доставка, мониторинг и подготовка к тестированию
**Цель**: Сделать систему готовой к внешнему тестированию и сбору метрик.

### Задачи:
1. [ ] Настроить доставку дайджестов:
   - Email интеграция (SMTP / SendGrid)
   - Или push-уведомления
   - Или встроенный inbox с уведомлениями
2. [ ] Реализовать сбор метрик:
   - D7/D30 retention
   - Open rate дайджестов
   - Reaction rate
   - Useful insight rate
   - Рост графа
3. [ ] Логирование событий для аналитики
4. [ ] Базовый дашборд для разработчика (мониторинг активности)
5. [ ] Документация для тестировщиков
6. [ ] Docker-compose для развёртывания
7. [ ] End-to-end тесты критических путей
8. [ ] Исправление багов, найденных при внутреннем тестировании

**Критерий готовности**: Система полностью автономна, доставляет инсайты пользователям, собирает метрики, готова к пилотному тестированию на реальной аудитории.

**Ожидаемое время**: 5-7 дней

## Технические решения (предварительные)

### Стек технологий:
- **Язык**: Python 3.11+
- **Графовая БД**: Neo4j (продакшн), NetworkX (для прототипирования)
- **LLM**: OpenAI API (GPT-4) или локальная модель через Ollama
- **Векторные эмбеддинги**: OpenAI embeddings или Sentence Transformers
- **Планировщик**: APScheduler (простота) или Celery + Redis (масштабируемость)
- **Бэкенд**: FastAPI
- **Фронтенд**: Минимальный React/Vue или HTMX для быстрого старта
- **Контейнеризация**: Docker + Docker Compose
- **Мониторинг**: Prometheus + Grafana (опционально для MVP)

### Структура репозитория (планируемая):
```
exocortex/
├── app/
│   ├── core/           # Ядро: модели графа, репозитории
│   ├── agents/         # Проактивный агент, анализ
│   ├── llm/            # LLM-пайплайны, промпты
│   ├── api/            # REST/GraphQL эндпоинты
│   ├── services/       # Бизнес-логика
│   └── utils/          # Утилиты
├── tests/              # Тесты
├── migrations/         # Миграции БД
├── docker/             # Docker конфигурации
├── docs/               # Документация
└── config/             # Конфигурация
```

## Риски и зависимости

1. **Зависимость от LLM API**: Качество извлечения сущностей и анализа напрямую влияет на полезность системы
2. **Сложность обнаружения противоречий**: Может потребоваться несколько итераций для настройки промптов
3. **Производительность графовых запросов**: При росте графа могут потребоваться оптимизации
4. **Персонализация**: Требует достаточного количества данных о пользователе для эффективности
